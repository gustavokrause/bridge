# 🌉 The fleet, in plain words

How to explain this system fast — pick the layer that fits your listener.

---

## One sentence

> It's an AI software team I run from one screen: I dump raw ideas in, it turns
> them into planned, risk-rated tasks, builds each one in isolation, and hands me
> a pull request to approve — and I control with a single dial how much it's
> allowed to ship on its own.

## The hook (lead with this)

> **I describe what I want in plain language; it comes back as merged code I approve.**

Unpack the rest only if they're curious.

## The analogy (sticks best)

Think of a small dev shop:

- **whale** is the *strategist* — takes messy notes and turns them into a
  sequenced backlog.
- **krill** is the *engineer* — picks up a task, writes the code on its own
  branch, reviews itself, opens a PR.
- the **personas** (ai-team) are the *specialists* it consults — product,
  backend, security, and more.
- **I'm the boss** who sets how much rope they get.

## 30-second version

> Two parts. **whale** is the brain: I drop in ideas, it distills them into
> concrete tasks, scores each by risk, and figures out the right order.
> **krill** is the hands: it runs each task in its own sandbox, has the AI review
> its own work, and produces a pull request. There's one **autonomy dial** — from
> "ask me at every step" to "just ship it" — and a hard rule that anything risky,
> or any change to the system itself, always stops for me. So I get a real team's
> throughput while keeping the brakes.

## If they push — what's actually hard here

The value isn't "AI writes code" (everyone has that). It's the **pipeline + the
brakes**:

```
idea → triage (risk-scored) → plan → build (isolated) → AI review → pull request → approve
```

with risk-gating and a **self-edit guard**, so it runs mostly unattended
*without* you losing control. That's the part that's hard to build, and the part
worth showing.
